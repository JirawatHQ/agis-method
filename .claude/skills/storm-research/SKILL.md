# STORM Research — วิจัยหลายมุมมอง + จับผิดตัวเอง

วิธีของ Stanford OVAL (NAACL 2024): **S**ynthesis of **T**opic **O**utlines through **R**etrieval and
**M**ulti-perspective **Q**uestion Asking. ผลวัด peer-reviewed: บทความ **จัดระเบียบดีกว่า baseline +25%**
และครอบคลุมกว้างกว่า +10%. หัวใจคือ **ถามหลายมุมบนแหล่งจริง แล้วให้แต่ละมุมจับผิดกันเอง** ก่อนสังเคราะห์

> อ้างอิงตัวจริง: [stanford-oval/storm], [hadufer/claude-storm] (skill ต้นทาง) — ดู context/wiki/research/storm_method

## เมื่อไหร่ควรใช้ / ไม่ควรใช้
- **ใช้:** คำถามที่ตอบมุมเดียวแล้วมีจุดบอด — ตัดสินใจสำคัญ, ชั่งข้อดีข้อเสีย, เปรียบเทียบทางเลือก, ประเด็นถกเถียง, ทำ dossier/รายงาน, ตรวจข้อเท็จจริงที่ต้องการหลายด้าน
- **ไม่ใช้ (แพงเกิน):** ถามข้อเท็จจริงเดียว, lookup นิยาม/ตัวเลข, งานที่ skill เฉพาะทางทำได้ตรงกว่า (หุ้น→`th-stock-analysis`, กฎหมายไทย→`ocs-law-lookup`, เพจปลอม→`fb-osint`) — ให้เรียก skill นั้นแทน หรือใช้ STORM ครอบโดยมอบ persona ให้ไปใช้ skill นั้นเป็นเครื่องมือ

## สองโหมด
| โหมด | เมื่อไหร่ | กลไก | ต้นทุน |
|------|---------|------|--------|
| **STORM-lite** (default) | คำถามวิเคราะห์/ตัดสินใจทั่วไป | ผู้ช่วย spawn subagent เอง (Agent tool) 5 persona ขนาน | ~6–9 agents |
| **STORM-full** | ผู้ใช้ opt-in: "ultracode" / "ใช้ workflow" / "เต็มรูปแบบ" / งานใหญ่มาก | `Workflow` tool — pipeline deterministic | ~12+ agents |

**ห้าม** ยิง Workflow เองถ้าผู้ใช้ไม่ opt-in — default คือ lite

## 5 มุมมอง (ปรับตามโดเมน — สำคัญ)
ค่าเริ่มต้น (หัวข้อทั่วไป): **Practitioner** (คนลงมือจริง) · **Academic** (ทฤษฎี/หลักฐานวิชาการ) ·
**Skeptic** (คนขี้สงสัย หาช่องพัง) · **Economist** (ต้นทุน/แรงจูงใจ/ตัวเลข) · **Historian** (เคยเกิดมาก่อน บทเรียนอดีต)

> **อย่ายึด 5 คนนี้ตายตัว** — จุดบอดคลาสสิกคือ "ทุกมุมนั่งเก้าอี้เดียวกัน". ปรับให้ครอบ stakeholder จริงของหัวข้อ:
> - **คดี/กฎหมาย:** พนักงานสอบสวน · อัยการ · ทนายจำเลย · ผู้เสียหาย · ผู้พิพากษา
> - **การเงินส่วนตัว:** นักวางแผนการเงิน · คนคุมความเสี่ยง · นักภาษี · คู่สมรส/ครอบครัว · ตัวคุณเองระยะยาว
> - **เทค/เว็บ/สินค้า:** dev · ผู้ใช้ปลายทาง · ทีมขาย · security · คู่แข่ง
> ก่อนเริ่มให้ประกาศ 5 persona ที่เลือก + เหตุผลสั้นๆ ว่าครอบใครบ้าง มีมุมไหนที่จงใจไม่ใส่

## Workflow (6 จังหวะ)

### 0) Frame + เลือก persona → **ประกาศ STORM banner (บังคับ)**
ตีกรอบคำถามให้คม → เลือก 5 persona ให้ครอบ stakeholder จริง → **แสดง banner ให้คุณเห็นก่อนลงมือทุกครั้ง** (เพื่อยืนยันว่า STORM ทำงานจริง ไม่ใช่ตอบมุมเดียวแล้วอ้าง). รูปแบบ:
```
🌀 STORM [lite|full] — "<คำถาม>"
Persona ที่ตั้ง (5):
1. <ชื่อมุม> — <ทำไมต้องมี / grounded ด้วยอะไร>
2. ... (ต้องมี Skeptic ≥1 + คนนอก ≥1)
มุมที่จงใจไม่ใส่: <ถ้ามี>
```
รอคุณดู 1 อึดใจ (ถ้าจะเพิ่ม/ตัด persona สั่งได้) แล้วค่อยเข้าขั้น 1. **ห้ามข้าม banner** — ถ้าไม่โชว์ persona = ถือว่าไม่ได้รัน STORM. ตอนจบให้ระบุด้วยว่ามุมไหน **มีส่วนจริง** ในผลลัพธ์ (persona ที่ค้นแล้วไม่เจออะไร = บอกตรงๆ)

### 1) Multi-perspective grounded research (ขนาน)
spawn subagent 1 ตัว/persona พร้อมกัน. **แต่ละ persona ต้อง grounded บนแหล่งจริง** (WebSearch/WebFetch; หรือ NotebookLM ถ้าเป็นเอกสารคุณ; หรือ skill เฉพาะทาง). สั่งให้ทำ interview ตัวเอง ~3 รอบ (ถาม→ค้น→ถามต่อจากคำตอบ) แล้วคืน:
- 3–6 finding เด่นในมุมของตน — แต่ละข้อ **มี citation (URL/มาตรา/เอกสาร)** ห้ามลอย
- ความมั่นใจของ finding (หลักฐานแน่/อ่อน) + ข้อที่ตัวเองยังไม่มั่นใจ

Prompt persona (ย่อ): *"คุณคือ [persona] วิเคราะห์ '[คำถาม]' จากมุมของคุณล้วนๆ. ค้นแหล่งจริงก่อนตอบ (ห้ามเดา). คืน finding 3–6 ข้อ แต่ละข้อมี citation + ระดับความมั่นใจ + จุดที่ยังไม่แน่ใจ. ชี้ประเด็นที่มุมอื่นน่าจะมองข้าม."*

### 2) Contradiction Map
รวม finding ทุก persona แล้วทำตาราง 3 กลุ่ม:
- **เห็นตรงกัน (≈จริง)** — หลายมุมยืนยัน หลักฐานแน่ → ความเชื่อมั่นสูง
- **ขัดแย้งกัน** — ใครว่าอะไร, ใครหลักฐานแน่/อ่อน, ทำไมถึงต่าง
- **ไม่มีใครแตะ (จุดบอด)** — ประเด็นที่ควรมีแต่ไม่มีมุมไหนพูดถึง

### 3) Synthesis
เรียบเรียงเป็นรายงานเดียว มี outline ชัด. ทุกข้ออ้างอิงต้องผูก citation จากขั้น 1

### 4) Adversarial peer review (จับผิดตัวเอง)
spawn subagent ชุดตรวจ (1–3 ตัว, โหมด full ให้ ≥3 โหวต) สั่งให้ **พยายามหักล้าง** ทุก claim + เช็ค citation ว่าจริง/ตรง:
- ถูก+แหล่งจริง → ยืนยัน
- ผิด/แหล่งไม่รองรับ/เกินจริง → แก้ หรือ **ตัดทิ้ง**
- default ถ้าไม่แน่ใจ = ถือว่ายังพิสูจน์ไม่ได้ (ลด score)

### 5) ออกรายงาน HTML
สร้างไฟล์ HTML (ฟอนต์ **Anuphan** — feedback_font_anuphan) มีครบ:
1. **สรุป 60 วินาที** — 3–5 บรรทัด decision-maker อ่านจบได้
2. **Findings จัดอันดับ + credibility score** — ตาราง: ข้อสรุป · คะแนน X/10 · มุมที่ **สนับสนุน** · มุมที่ **แย้ง** · หลักฐาน
3. **Contradiction Map** จากขั้น 2
4. **มุมที่ยังขาด** — จุดบอด + persona ที่ควรเพิ่มรอบหน้า
5. **References** — คลิกได้ dedup URL/มาตรา/เอกสาร
บันทึกที่ `context/rawdata/research/storm/storm-<slug>-YYYY-MM-DD.html` แล้วบอก path + สรุป 60 วิ ใน chat

## Credibility score (เกณฑ์)
9–10 หลายมุมยืนยัน+หลักฐานปฐมภูมิ · 7–8 หลักฐานดีแต่บางมุมแย้ง · 5–6 หลักฐานผสม/ยังถกเถียง ·
≤4 หลักฐานอ่อน/แหล่งเดียว/ผ่าน review แล้วสั่นคลอน (คงไว้เฉพาะถ้าจำเป็น + ติดธงชัด)

## STORM-full (Workflow) — โครงที่ควรเขียน
pipeline: personas → `parallel` grounded-research/persona → barrier: dedup + Contradiction Map → synthesis →
`parallel` adversarial-verify/claim (≥3 โหวต, majority refute = ตัด) → เขียน HTML. ดู pattern "adversarial verify"
+ "loop-until-dry" ในสเปค Workflow tool. เขียน script → รันเมื่อ user opt-in เท่านั้น

## Anti-patterns
- persona ที่ไม่ค้นแหล่ง = แค่ role-play → ผิดหัวใจ STORM (ต้อง grounded)
- ข้าม adversarial review เพราะ "ดูน่าเชื่อแล้ว" → นี่คือขั้นที่กันข้อมูลผิดหลุด
- ยึด 5 persona เดิมทุกหัวข้อ → จุดบอดเก้าอี้เดียวกัน
- credibility score โดยไม่ระบุใครสนับสนุน/แย้ง → ตัวเลขลอยไม่มีประโยชน์
